<objective>
Prove the new CI can go RED and lock it in as a merge gate. Delivers CI-02: (1) open a
throwaway PR that reverts a real guard (`test_drift`), observe `ci-gate` fail in the GitHub
Actions UI itself, then close the PR unmerged and delete the branch so nothing lands on main;
(2) register `ci-gate` as the sole required status check on `main`'s branch protection via
`gh api`, with strict mode on, require-PR (0 approvals), and include-administrators.

Purpose: A workflow that merely runs satisfies "CI exists," not "CI enforces." This plan
applies the milestone's own standard to CI itself — a guard is not shipped until it has been
observed failing — and closes the retrospective's dominant failure mode (merging on a local
green) by hard-blocking direct-to-main pushes and un-gated merges.

Output: A RED-run evidence link and the branch-protection API state. No committed files change
(branch protection is imperative repo-admin state, not a file). This plan is NOT autonomous:
it requires the workflow to already be live on GitHub, human observation of the red run in the
Actions UI, and a repo-admin `gh api` call.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → GitHub admin API | The branch-protection PUT uses the operator's locally-authenticated `gh` token (repo-admin scope). This credential is never committed, never echoed into the workflow, and never leaves the operator's machine. |
| throwaway PR → main | The RED-proof PR is deliberately never merged; the mutation is confined to a branch that is deleted. `enforce_admins` + require-PR ensure even an accidental merge attempt is gated. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-05 | Elevation of Privilege | Branch-protection PUT via admin `gh` token | accept | The call uses the operator's existing local `gh` auth (repo-admin), run interactively as a human-gated checkpoint. No new credential is created, stored, or committed. This is a one-shot config change, verified by an immediate GET. |
| T-10-06 | Tampering | The RED-proof mutation reaching main | mitigate | The drift-reverting change lives only on a throwaway branch whose PR is closed unmerged and whose branch is deleted; acceptance explicitly verifies `origin/main` never receives the commit. The very protection rule this plan installs (`enforce_admins`, require-PR) also blocks accidental direct-to-main landing going forward. |
| T-10-07 | Information Disclosure | Evidence links / GET output | accept | The captured artifacts are a public Actions run URL and the branch-protection JSON (repo config, no secrets). Nothing sensitive is disclosed. |
| T-10-SC | Tampering | Package/dependency installs | n/a | This plan installs nothing (no npm/pip/cargo). No package-legitimacy gate applies. |
