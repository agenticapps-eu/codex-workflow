## Summary

This phase closed CR-01 (decoupled strip entry/exit) and CR-02 (unanchored `PROV_RE`) in
`migrations/0009-spec-11-region-aware-placement.md`, and the suite's own claim of
345 PASS / 0 FAIL / 1 SKIP holds — I re-ran `bash migrations/run-tests.sh 0009` and got
63 PASS / 0 FAIL against the current HEAD, unmodified. The fixes for the fixture shapes
that were reproduced (13/14/15, drifted/orphaned/mixed provenance) are real and I could not
find a gap in the `unresolved`/`END{exit 4}` guard, the refuse-gate `elif` pair, or the
h2-count backstop for any of the shapes those fixtures cover.

However, the anchoring fix (`^...$` on all four `PROV_RE` sites, CR-02) only closes the
*substring-mention* hazard. It does not close two other input shapes that reach the same
underlying defect class (decoupled/mismatched anchor vs. terminator) through different
doors, and I reproduced both empirically by extracting the **actual, unmodified** Step 1
Apply block from `migrations/0009-spec-11-region-aware-placement.md` at HEAD and running it
against crafted fixtures in a scratch directory (no tracked files were touched):

1. **CRLF line endings** defeat every `$`-anchored regex in the awk layer (the strip's entry
   regex, the strip's terminator, and the insert's region-marker anchor), while the
   non-anchored `/^## /` alternative is unaffected — reproducing the *exact* pre-0009 defect
   (block inserted inside the GitNexus-managed region) while the migration reports success.
2. **An exact, whole-line quotation of the provenance marker and/or the §11 heading inside
   prose or a markdown code fence** (e.g. a troubleshooting note that shows the operator what
   a healthy block looks like) is not a "mention" in the CR-02 sense — it *is* a real anchored
   match on both regexes simultaneously — and it makes the strip latch on the fenced example
   and swallow real, unrelated user content up to the next incidental `## ` heading.

Both are documented as open, unaddressed hazards in upstream's own "Known limitations"
section (cited verbatim in this phase's own ADR, `docs/decisions/0010:548-549`, while filing
the CR-01 issue against upstream) — so the team was aware of the *class* of risk, but neither
this migration document nor its test suite carries any disclosure or fixture for it, and I
have now confirmed both are live against the *fixed*, current 0009.

`run-tests.sh` itself is sound at `standard` depth for the shapes it does assert — I found no
assertion in `test_migration_0009` or `test_drift` that is trivially satisfiable or is
checking the wrong layer. One stale comment (Info) survives from a prior plan's own recorded
housekeeping note.

## Critical Issues

### CR-01: CRLF line endings defeat every `$`-anchored anchor/terminator regex, reinserting the §11 block inside the GitNexus-managed region

**File:** `migrations/0009-spec-11-region-aware-placement.md:207` (`PROV_RE` definition),
`:290` (strip entry), `:304` (strip terminator alternation), `:387` (insert anchor
alternation)

**Issue:** The strip terminator and the insert anchor both alternate on
`/^## / || /^<!-- gitnexus:start -->$/`. The second branch is `$`-anchored. Standard POSIX
`awk` (verified against the system `awk` — this is not implementation-specific behavior;
`gawk`/`mawk`/BWK `awk` all treat `\r` as an ordinary trailing character, not something `$`
consumes) does **not** special-case `\r\n` line endings: a line `<!-- gitnexus:start -->\r\n`
does **not** match `/^<!-- gitnexus:start -->$/`, because the `\r` sits between the matched
text and the record's true end. The `/^## /` alternative has no `$` anchor and matches
regardless of a trailing `\r`. This asymmetry breaks the "anchor and terminator are one
decision" invariant the migration's own prose (`:430-440`) calls "the highest-severity
mechanic in this migration" — but only for CRLF input, which the invariant's proof never
considered.

**Concrete triggering input:** A **region-led** `AGENTS.md` (the exact shape `01-gitnexus-led-inject`
tests) saved with CRLF line endings — e.g. checked out on a project with
`core.autocrlf=true`, or edited on Windows, or produced by any tool that writes `\r\n`.
No `§11` block present yet (State C).

**Empirical result** (extracted the real `**Apply:**` fenced block from
`migrations/0009-spec-11-region-aware-placement.md` at HEAD, ran it via `eval` against a
CRLF-converted copy of the `01-gitnexus-led-inject` fixture, in a scratch dir):

```
=== rc=0 ===
INFO: migration 0009 Step 1 — §11 block anchored above any managed region.
```

but the resulting file has:
```
provenance line: 10
gitnexus:start line: 5
```

i.e. the provenance/§11 block landed **below** (inside) `<!-- gitnexus:start -->`, not above
it — the file after Apply:

```
<!-- gitnexus:start -->
# GitNexus — Code Intelligence

This project is indexed by GitNexus as **demo** (100 symbols).

<!-- spec-source: agenticapps-workflow-core@0.4.0 §11 -->
## Coding Discipline (NON-NEGOTIABLE)
...
## Always Do
- MUST run impact analysis before editing any symbol.

## Never Do
- NEVER rename symbols with find-and-replace.
<!-- gitnexus:end -->
```

**Consequence:** This is the *exact* defect this migration exists to fix (see the document's
own opening paragraph: a block landing inside `<!-- gitnexus:start -->…<!-- gitnexus:end -->`
"is regenerated and destroyed with no diagnostic" on the next `gitnexus analyze`) — reproduced
against the fixed 0009, on CRLF input, while the migration prints an INFO success message
that is the literal opposite of what happened. No content is destroyed on *this* run (the
region markers stay paired), but the freshly-vendored §11 block is now living inside a region
that will be silently regenerated and destroyed by the very next `gitnexus analyze`, with zero
diagnostic from either tool.

**Fix:** Normalize line endings before any structural matching (e.g.
`awk '{ sub(/\r$/, ""); print }'` into a working copy used for scanning, or strip/track a
per-line `\r` flag and re-append it on output), or explicitly document CRLF as an
out-of-scope precondition and add a pre-flight guard that refuses on any `\r`-terminated line
in `AGENTS.md` (fail closed rather than silently mis-anchoring). Add a fixture exercising a
CRLF-encoded `01-gitnexus-led-inject`-shaped file to the suite either way — none of the ten+
cases currently vary line-ending style.

### CR-02: An exact, whole-line quotation of the provenance marker or §11 heading (e.g. inside a markdown code fence documenting the block) is not a "mention" — it latches the strip and silently deletes real, unrelated content

**File:** `migrations/0009-spec-11-region-aware-placement.md:288-324` (strip awk),
`:216-255` (refuse-gate `elif` pair)

**Issue:** CR-02's fix anchors `PROV_RE` (`^...$`) specifically to stop *substring* mentions
(a marker referenced inside a longer line, e.g. inside an HTML comment as prose) from
false-triggering the strip's entry condition. Fixtures 07 and 11 both test that shape and
correctly pass. But an anchored regex still matches a line that is *exactly* the provenance
comment or *exactly* the §11 heading, wherever that line physically sits in the file —
including inside a markdown code fence used to document or illustrate the block (a
realistic authoring pattern: a troubleshooting note explaining "here's what a healthy §11
marker looks like"). Neither refuse-gate `elif` catches this, because both are file-global
presence checks (`grep -q heading && ! grep -qE PROV_RE`, and the inverse) — a fenced example
that quotes *both* strings together satisfies "heading present" and "provenance present"
simultaneously, so neither conjunction is true and both refuse gates stay silent, identical
to the blind spot fixture 15 was written to catch for a *second real block* — but here the
trigger is a fenced *example*, not a second real block, and none of the shipped fixtures test
this shape.

**Concrete triggering input:**
```markdown
# My Project

## Troubleshooting

If your AGENTS.md ever needs the §11 marker restored by hand, it looks
exactly like this:

```
<!-- spec-source: agenticapps-workflow-core@0.4.0 §11 -->
## Coding Discipline (NON-NEGOTIABLE)
```

Do not delete this troubleshooting note.

## Deployment
Real deployment content that must survive.
```

**Empirical result** (same extraction method, real Apply block, real mirror content, run
against the above verbatim):

```
=== rc=0 ===
INFO: migration 0009 Step 1 — §11 block anchored above any managed region.
=== byte-identical to before? ===
**CHANGED**
```

After-state:
```
# My Project

<!-- spec-source: agenticapps-workflow-core@0.4.0 §11 -->
## Coding Discipline (NON-NEGOTIABLE)

Some prose.

### 4. Goal-Driven Execution

more prose

## Troubleshooting

If your AGENTS.md ever needs the §11 marker restored by hand, it looks
exactly like this:

```
## Deployment
Real deployment content that must survive.
```

The strip entered `in_block` at the fenced example's provenance line, swallowed the fenced
example's own heading line (`swallowed_own_h2=1`), then consumed every subsequent line —
including the closing code fence, the sentence "Do not delete this troubleshooting note.",
and the blank line before it — because none of them matched the `## `/marker terminator,
until it reached the unrelated `## Deployment` heading and stopped there. The §11 block
itself gets correctly re-vendored at the top (that part is not the bug), but real,
user-authored content between the fenced example and the next real heading is **silently
deleted**, and the migration reports success.

**Consequence:** Genuine, silent data loss on a realistic authoring pattern (a
troubleshooting/reference note quoting the exact marker), reachable against the fully-fixed
0009 — the same defect class (decoupled entry vs. terminator) the whole phase exists to
close, via a trigger shape none of fixtures 07/11/13/14/15 cover (those test either
non-matching substrings or genuine second *structural* blocks, never an exact quotation
embedded in surrounding prose).

**Fix:** Track fenced code-block state (paired ``` / ~~~ delimiters) in the strip/refuse-gate
awk and skip matching inside fences; or, at minimum, add this as a documented "Known
limitations" entry in 0009 itself (it is currently undocumented there — see WR-02 below) and
add a refusal path when the exact heading appears without a directly-preceding blank line +
provenance pair immediately above it (a weaker but strictly-safer heuristic than the current
"anywhere in file" presence check).

## Warnings

### WR-01: Zero-byte `AGENTS.md` hard-aborts and permanently strands the project below `to_version`, contradicting the migration's own stated design principle

**File:** `migrations/0009-spec-11-region-aware-placement.md:210-215` (the
missing-`AGENTS.md` skip path), `:344-348` (the strip-produced-no-output abort)

**Issue:** The pre-flight section (`:146-154`) and the `04-no-agentsmd` case both establish,
in the document's own words, that an *abort* on a missing `AGENTS.md` would be
"unrecoverable": Step 2 never records `0.7.0`, so 0009 stays pending forever and every future
migration (`0010+`) never becomes pending either — hence the deliberate informational-skip
design. A **zero-byte** `AGENTS.md` (e.g. from `touch AGENTS.md`, an interrupted write, or a
prior tool crash) is a materially identical "nothing to heal" state, but it is not routed
through the `[ ! -f AGENTS.md ]` skip branch — `test -f` passes on an empty file. It falls
through to the `else` branch, the strip awk runs on zero input, produces zero-byte output, and
hits the `[ ! -s AGENTS.md.0009.strip ]` guard at `:344`, which `exit 3`s with
"the strip pass produced no output; refusing to replace AGENTS.md with a possibly-truncated
result."

**Concrete triggering input:** `AGENTS.md` present, zero bytes.

**Empirical result:** `rc=3`, file left untouched (0 bytes, unchanged) — reproduced against
the real Apply block.

**Consequence:** Not data loss (there was nothing in the file to lose), but exactly the
"unrecoverable, stranded below `to_version` permanently" outcome the migration's own design
principle says must be avoided via a skip rather than an abort. The diagnostic text is also
misleading for this specific case — "possibly-truncated result" implies the strip *removed*
something, when in fact the input was already empty before the strip ran.

**Fix:** Extend the missing-`AGENTS.md` skip condition to `[ ! -s AGENTS.md ]` (matching the
pre-flight's own `test -s` philosophy applied to the mirror), or at minimum special-case
zero-byte input in the strip-output guard with a distinct, accurate diagnostic
("AGENTS.md was already empty before Step 1 ran; nothing to heal") rather than routing it
through the generic "possibly-truncated" abort message.

### WR-02: CRLF and fenced/quoted-marker hazards are known (upstream-documented) but undisclosed in this repo's own 0009 or its "Notes"/limitations section

**File:** `migrations/0009-spec-11-region-aware-placement.md` (no "Known limitations"
section covers this); `docs/decisions/0010-region-aware-spec-11-placement.md:548-549`

**Issue:** The ADR records, in the course of filing the upstream CR-01 issue, that
upstream's own "Known limitations" section (`f9354cc:0029:411-420`) already lists CRLF and
fenced-code-block markers as unresolved hazards in the *same* strip mechanism this migration
ports. The team therefore had direct knowledge of this hazard class while writing 0009's own
"Notes" section (`:499-521`, which documents the `PROV_RE` anchoring precedent in detail) —
yet 0009 carries no equivalent disclosure, and the phase's fixture set (07/11/13/14/15,
09.1-04 through 09.1-06) never exercises either shape, despite this being exactly the same
class of defect (decoupled entry/exit) this phase's whole mandate was to close.

**Consequence:** A future reader has no signal that these are open, load-bearing risks rather
than closed ones — the document's own framing ("PROV_RE anchoring... closes CR-02", `:501`)
reads as if the marker-matching hazard class is fully resolved, when CR-01 and CR-02 above
show it is not.

**Fix:** Add a "Known limitations" entry to 0009 itself (mirroring the existing "Awk
portability (A2)" entry at `:154`) naming CRLF and exact-quotation-in-prose/fence as open,
unaddressed hazards, with a pointer to the upstream issue thread for CRLF if one gets filed.
