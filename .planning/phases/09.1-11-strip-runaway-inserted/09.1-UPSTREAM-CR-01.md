## Suggested issue title

`migration 0029: strip runaway when provenance is present but the exact §11 heading is drifted/absent (CR-01)`

## Suggested issue body

### Summary

`migrations/0029-region-aware-spec-11-placement.md`'s Step 1 Apply couples the
managed block's **entry** condition to an anchored provenance marker, but its
**exit** condition is gated behind a separate flag (`swallowed_own_h2`) that is
only ever set by an *exact* match on `## Coding Discipline (NON-NEGOTIABLE)`.
When a provenance line is present but the heading that should immediately
follow it is drifted, renamed, or otherwise not an exact match, the strip
awk's `in_block` state latches at `1` and never un-latches — `in_block { next
}` then consumes every remaining line in the file to EOF.

This is still live at `f9354cc:migrations/0029-region-aware-spec-11-placement.md:222-241`,
the same shape acknowledged fixed for the *entry* anchoring (CR-02, closed by
this repo's own PR #89) but not for the *exit* coupling. Anchoring `PROV_RE`
narrows **how you get in**; it does nothing about **not being able to get
out**.

### Reachability argument (the load-bearing part)

The Step 1 Apply's existing conflict/abort branch is:

```bash
elif grep -q '^## Coding Discipline (NON-NEGOTIABLE)$' CLAUDE.md \
     && ! grep -qE "$PROV_RE" CLAUDE.md; then
  echo "ABORT: CLAUDE.md contains a '## Coding Discipline (NON-NEGOTIABLE)'"
  ...
  exit 3
```

That fires on **heading present AND provenance absent** — the hand-paste
conflict case. This defect needs the **inverse**: provenance present, heading
absent or drifted. That state does not match the conflict branch's condition,
so it falls straight through the `elif` into the `else` and into the strip
pass unchanged. This is the detail a reviewer needs to not dismiss "drifted
heading" as an exotic input — the existing guard's condition is the *literal
inverse* of what would need to fire to catch it, and nothing else in the Apply
block checks for it before the strip runs.

Reachable via, at minimum:
- a hand-edited or otherwise drifted/renamed `## Coding Discipline
  (NON-NEGOTIABLE)` heading;
- provenance left over from an earlier migration state, at a version the
  current `PROV_RE` still matches (it matches any version via
  `@[^[:space:]]+`) but whose heading no longer matches exactly;
- the orphaned-provenance state this migration document's own Step 1 comment
  already admits it can produce (`f9354cc:0029:252`: *"yet awk still exits 0
  with non-empty output (the rest of the file, plus an orphaned provenance
  line)"* — that comment describes a zero-byte `$SPEC_BLOCK` edge case in the
  *insert* pass, but the resulting file shape — an orphaned provenance line
  with no following heading — is exactly the shape that re-enters this same
  runaway on the *next* run of Step 1's strip).

### Why all three post-strip guards pass

1. `[ -s CLAUDE.md.0029.strip ]` — passes; the truncated output is non-empty.
2. The insert pass's own shape assertion
   (`grep -q '^## Coding Discipline (NON-NEGOTIABLE)$' CLAUDE.md.0029.tmp`)
   — passes, **because the insert pass that runs immediately after the strip
   re-adds the exact heading this guard checks for.** A guard meant to detect
   a bad strip is satisfied by the pass that follows the bad one, not by the
   strip's own correctness.
3. Version/frontmatter post-checks are unrelated to content loss and do not
   see it either.

`mv` then commits the truncated file over the user's `CLAUDE.md`.

### Acknowledging PR #89

This repo's own PR #89 (squash-merged as `f9354cc`) already closed CR-02 —
anchoring `PROV_RE` at all its sites (`:142`, `:146`, `:175`), plus the
`11-prose-mention-provenance` fixture. That fix is real and closes the
substring-match hazard on the *entry* side. **It does not close this issue.**
Anchoring narrows how the block is entered; the runaway above is about the
block never being *exited* once entered, which is a property of the
`swallowed_own_h2`-gated exit rule, not the entry regex.

### Known limitations section omits this

`f9354cc:migrations/0029-region-aware-spec-11-placement.md:411-420` carries a
**"Known limitations (no live instance, not fixed by design)"** section
listing two limitations: a CRLF `CLAUDE.md` defeating the anchored regexes,
and a marker written literally inside a fenced code block. **This runaway is
not in that list.** That is offered as evidence this is new information to
you, not something already triaged and deliberately accepted — if it is in
fact already tracked elsewhere, apologies for the noise, and a pointer to
where would be appreciated.

### Suggested fix (offered, not demanded)

We hit the identical defect in our own downstream port
(`codex-workflow`, migration `0009`, ported from your `8520f90`) and closed it
with two changes, offered here only as one way to do it — happy to be
overruled on approach:

1. A pre-surgery refuse gate whose condition is the literal inverse of the
   existing hand-paste-conflict branch: `provenance present AND exact heading
   absent` ⇒ abort `exit 3` before any file surgery, file left byte-identical.
   This closes the file-global case (a single block whose heading drifted, or
   an orphaned provenance line with no block).
2. A fail-closed guard inside the strip awk itself, for the one shape the
   file-global gate above cannot see (a *second*, later block in the same
   file with a drifted heading, where an *earlier* block's healthy pair
   already satisfies the file-global predicate): an `unresolved` flag set
   when a second provenance line arrives while a prior one never found its
   heading, checked in `END { if (unresolved || (in_block &&
   !swallowed_own_h2)) exit N }`.

We considered and rejected un-latching the strip's entry condition and
inserting anyway (reset `in_block` at any structural boundary, not only a
swallowed one) — it avoids data loss but silently leaves two similar `##
Coding Discipline` headings in the file, and in our testing it still ran away
to EOF on the orphaned-provenance shape specifically (no heading at all
between the provenance line and EOF), so it did not even fully close the
runaway on its own.
