<objective>
Close 08-REVIEW.md WR-02: migration 0008 Step 3 validates the bindings-table
header but then inserts the `plan-review` row after the FIRST `|---` line found
anywhere in `AGENTS.md`. The two are uncorrelated, so a target repo whose
AGENTS.md holds any other Markdown table before the bindings table gets the row
inserted into that unrelated table instead.

Purpose: this was code-inspection-only in the review ("I could not construct a
failing case against this repo's own AGENTS.md"). It was **reproduced during
gap-closure planning** — see the exact fixture in `<interfaces>`. It is a real
silent-corruption risk in OTHER people's repos, and it is self-sealing: Step 3's
idempotency check is `grep -q '^| plan-review' AGENTS.md`, which would find the
misplaced row on any later re-run and mark the step already-applied, permanently
masking that the real bindings table never got the row. A repo in that state
ships a bindings table missing the very gate this phase exists to bind, and the
migration reports success. The failure class is the one the phase exists to
close, in the phase's own migration.

Output: an insertion correlated with the header the step already validates,
plus the fixture the existing suite lacks.
</objective>

**The defect lives in two places and must be fixed in both** — the awk is
duplicated verbatim:
- `migrations/0008-plan-review-gate.md:287-297` — the documented Step 3 pass
  (the source of truth; operates on `AGENTS.md`).
- `migrations/run-tests.sh:1143-1153` — the test's mirrored copy (operates on
  `AGENTS.md.scope-shaped`). Fixing only the doc leaves the test green against
  stale logic; fixing only the test makes the suite lie about the migration.

The offending rule, identical in both:
`/^\|---/ && !ins_pr { print; print pr; ins_pr=1; next }`

The other three rules in the same pass (brainstorm split, tdd collapse,
passthrough) are correct and match on their own specific row text — leave them
alone.

Step 3 already computes, at `0008-plan-review-gate.md:261-271`, a validated
header and ABORTS with exit 7 on mismatch:
`TPL_HEADER="$(grep -m1 '^| Gate |' "$TPL")"` /
`TARGET_HEADER="$(grep -m1 '^| Gate |' AGENTS.md)"`.
So the header is already known-good at insertion time — it is simply not
correlated with the separator the awk matches. That correlation is the fix.

**Reproduced during planning.** Given an AGENTS.md holding an unrelated
`| Tool | Purpose |` table before the `| Gate |` bindings table, the current
pass inserts `| plan-review | ... |` directly beneath the *Tool/Purpose*
separator, leaves the bindings table untouched, and exits 0. Adding a
`/^\| Gate \|/ { seen_hdr=1 }` rule and gating the separator match on
`seen_hdr` puts the row in the correct table and leaves the unrelated table
byte-identical. Both behaviors were confirmed by direct execution.

**Existing test-harness contracts** (`test_migration_0008` starts at line 777):
- `_table_data_rows <file>` — used at 1127 and 1161-1164 to count data rows and
  distinct gates.
- `assert_check <label> <predicate> <dir> <applied|not-applied>` — the
  idempotency-predicate helper.
- `extract_to` — pulls reference states out of git.
- The Step 3 fixture is `$tmp/AGENTS.md.scope-shaped`, a standalone table-only
  file that STARTS with the table and has no preceding content — which is
  exactly why it cannot catch WR-02.
- The fixture has a self-guard at 1126-1135 pinning it to 15 rows / 1
  brainstorm row. Do not disturb it; add a NEW fixture instead.
- Post-conditions asserted at 1166-1179: row count == 16 AND distinct-gate
  count == 16.
