<table_migration>
## Migrated installs must match fresh ones — the table, not just the ritual

Review's fourth HIGH against the original plan (08-REVIEWS.md, Codex): migration
0008 as first planned inserts the ritual section but never adds the `plan-review`
row or collapses the duplicate `tdd` row in an existing project's `AGENTS.md`.
Fresh installs get the corrected table from the template; migrated installs do
not. **Fresh and migrated installs therefore diverge** — which is precisely the
class of failure D-19's single-source rule exists to prevent, and it is the 0007
lesson re-learned. Adopted: 0008 gains a table step, sourced from the same
template.

**Which table shape a downstream install has.** This repo's own `AGENTS.md` uses
`| Gate | Bound skill | Applies to scaffolder? |`, while the template
`agents-md-additions.md` uses `| Gate | Bound skill | Scope |`. That asymmetry is
**local to this repo** — this repo's `AGENTS.md` is hand-maintained and predates
the template. A downstream install's `AGENTS.md` table was *written by the setup
skill from `agents-md-additions.md`*, so it carries the `Scope` shape and the
template's rows verbatim. That is what makes template extraction the right source
here, exactly as it is for the prose.

**Be defensive about it anyway. Do not assume; detect.** If the target's table
header does not match the template's header line, **skip the table step with a
loud warning and a distinct note in the post-checks** rather than guessing which
column means what. A migration that mangles a table it did not understand is
worse than one that declines to touch it and says so — the operator can fix a
skipped step, but a corrupted table is silent. Record this as a documented skip
case, per 0007's `## Skip cases` convention.

**Do NOT touch this repo's own `AGENTS.md` from this plan.** Plan `08-04` already
corrected it by hand, in its own column shape. This plan writes the migration
step that does it for *other* repos, and tests that step against synthetic
fixtures. `AGENTS.md` is not in this plan's `files_modified` — and note that this
repo's own hand-maintained table is exactly the wrong-shape target the header
guard is built to decline.
</table_migration>
