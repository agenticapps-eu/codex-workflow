<table_migration>
## Migrated installs must match fresh ones — the table, not just the ritual

Review's fourth HIGH against the original plan (08-REVIEWS.md, Codex): migration
0008 as first planned inserts the ritual section but never adds the `plan-review`
row or collapses the duplicate `tdd` row in an existing project's `AGENTS.md`.
Fresh installs get the corrected table from the template; migrated installs do
not. **Fresh and migrated installs therefore diverge** — which is precisely the
class of failure D-19's single-source rule exists to prevent, and it is the 0007
lesson re-learned. Adopted: 0008 gains a table step, sourced from the same
template.

**Which table shape a downstream install has.** This repo's own `AGENTS.md` uses
`| Gate | Bound skill | Applies to scaffolder? |`, while the template
`agents-md-additions.md` uses `| Gate | Bound skill | Scope |`. That asymmetry is
**local to this repo** — this repo's `AGENTS.md` is hand-maintained and predates
the template. A downstream install's `AGENTS.md` table was *written by the setup
skill from `agents-md-additions.md`*, so it carries the `Scope` shape and the
template's rows verbatim. That is what makes template extraction the right source
here, exactly as it is for the prose.

**The asymmetry is intentional and pre-existing — do NOT "fix" it by harmonizing
the columns.** The two tables cannot be byte-identical, because their third
columns answer different questions: the template's `Scope` says *when in the
lifecycle the gate fires*; this repo's `Applies to scaffolder?` says *whether this
particular repo is subject to it* (`No (no UI)`, `Yes (always)`). Both are
correct for their audience. Round 2's reviewers both implicitly assumed one table;
they are two. The wrong-shape skip case below **depends** on this repo's shape
being the one a downstream install must NOT have — harmonizing the columns would
delete the only realistic wrong-shape fixture in the repo.

## The brainstorm-row divergence — round 2's top consensus concern, and the fix

Codex and OpenCode independently reached the same defect from different
directions, with matching arithmetic. It is real. **The operator's adjudication:
keep `08-04`'s Part A2 and extend this plan's Step 3 to mirror it.** Do not
re-litigate; implement.

**The verified facts** (checked against the real files — the reviewers were
working from plan text and got some of this wrong, so use these numbers):

| File | Header | Data rows | brainstorm | `\| tdd` rows | plan-review |
|---|---|---|---|---|---|
| template `agents-md-additions.md` | `\| Gate \| Bound skill \| Scope \|` | **15** | **COMBINED** in one row | 2 | absent |
| this repo's `AGENTS.md` | `\| Gate \| Bound skill \| Applies to scaffolder? \|` | **16** | already SPLIT | 2 | absent |

So a real v0.5.0 downstream install — whose table the setup skill wrote from the
template — carries the template's shape: **15 rows, combined brainstorm**.

**The defect.** Plan `08-04` Part A2 splits the template's combined brainstorm
row. An earlier revision of *this* plan's Step 3 applied only the `tdd` collapse
and the `plan-review` add, and its Apply text said *"Leave every other row
untouched"* — so the two paths end in different shapes.

Decompose the template's 15 pre-0008 rows as **1 combined-brainstorm + 2 tdd + 12
others** (= 15 rows; and 2 + 1 + 12 = **15 distinct gates**, since the combined row
names two gates and the two `tdd` rows name one). Then:

| Path | brainstorm | tdd | others | plan-review | **rows** | **gates** |
|---|---|---|---|---|---|---|
| a real v0.5.0 install, pre-migration | 1 row (2 gates) | 2 rows (1 gate) | 12 | 0 | **15** | **15** |
| …after the round-1 Step 3 (collapse `tdd`, add `plan-review`) | 1 row (2 gates) | 1 | 12 | 1 | **15** | **16** |
| a FRESH install after `08-04` (split, collapse, add) | 2 rows (2 gates) | 1 | 12 | 1 | **16** | **16** |

**15 rows vs 16 rows.** Migrated and fresh disagree on row shape — exactly the
D-19/D-20 divergence class this phase exists to close. And Task 1's assertion
*"the row count equals the template's row count"* (16) becomes **unachievable
against a representative fixture**: it passes only if the fixture is built with an
already-split brainstorm row, which no real install has. The test would be
unrealistic-and-passing — false confidence, which is worse than a red test.

**The fix — Step 3 applies ALL THREE corrections, every row template-sourced.**

    1. SPLIT the combined `| brainstorm-ui / brainstorm-architecture |` row into
       two rows — `brainstorm-ui` and `brainstorm-architecture`, both bound to
       `superpowers:brainstorming`, both scoped `pre-phase`.
    2. COLLAPSE the duplicate `tdd` rows to the template's single row.
    3. ADD the `plan-review` row as the first data row.

    15 → split brainstorm (16) → collapse tdd (15) → add plan-review (**16**)
    = 16 rows / 16 distinct gates — identical to a fresh install. ✓

All three take their rows from the **template**, via the same extraction shape
Step 3 already uses for `plan-review` and `tdd` — **never a heredoc'd literal**
(D-19, single source of truth). The brainstorm rows are not special-cased: they
are two more rows read out of the same table. `config-hooks.json` models them as
two separate gates (`pre_phase.brainstorm_ui` and `pre_phase.brainstorm_architecture`
— verified), which is the same authority Part A2 rests on and the same authority
the `tdd` collapse rests on in the other direction.

**After Step 3, row count == distinct-gate count in every file**, and a migrated
v0.5.0 install reads the same 16/16 as a fresh one. That makes this plan's
`must_haves` claim — *"a migrated install reaches the same bound state as a fresh
one"* — literally true. It was misleading as written; it is now verifiable, and
Task 1 verifies it.

**Be defensive about it anyway. Do not assume; detect.** If the target's table
header does not match the template's header line, **skip the table step with a
loud warning and a distinct note in the post-checks** rather than guessing which
column means what. A migration that mangles a table it did not understand is
worse than one that declines to touch it and says so — the operator can fix a
skipped step, but a corrupted table is silent. Record this as a documented skip
case, per 0007's `## Skip cases` convention.

**Do NOT touch this repo's own `AGENTS.md` from this plan.** Plan `08-04` already
corrected it by hand, in its own column shape. This plan writes the migration
step that does it for *other* repos, and tests that step against synthetic
fixtures. `AGENTS.md` is not in this plan's `files_modified` — and note that this
repo's own hand-maintained table is exactly the wrong-shape target the header
guard is built to decline.
