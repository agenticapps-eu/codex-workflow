`verifier` is a NEW key no other gate carries. That is intentional per D-01's
hybrid decision: every other gate is purely declarative (agent-read); this one
alone carries a programmatic-verifier pointer. ADR-0009 (plan `08-03`) records why.

**The bare `${CODEX_HOME}` form in the config is correct — do not "normalize" it.**
Review flagged that the config uses `${CODEX_HOME}/...` while executable
instructions elsewhere use `${CODEX_HOME:-$HOME/.codex}/...` (08-REVIEWS.md,
Codex, MEDIUM). This asymmetry is deliberate and stays: per threat T-08-17 the
config's `verifier` value is **documentation of where the verifier lives, not a
string anything executes** — nothing shell-expands it, so a shell default has no
meaning there, and every installed-path reference in this repo's config-shaped
files uses the bare form (08-PATTERNS.md §"`${CODEX_HOME:-$HOME/.codex}`
path-resolution idiom" documents the split). The **ritual prose** — which IS an
executable instruction — uses the fallback form, and Task 2 requires it.

**Existing config shape** — `hooks` has four groups: `pre_phase`, `per_task`,
`post_phase`, `finishing`, each a flat object keyed by gate-slug, each gate
object carrying `skill`, `fires_when`, and gate-specific fields. `pre_execution`
becomes a fifth group, inserted as the first key of `hooks` (trigger-order
readability; JSON key order is not semantically significant to any consumer in
this repo).
