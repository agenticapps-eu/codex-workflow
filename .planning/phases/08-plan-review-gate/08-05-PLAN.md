**Config target differs from 0007's.** 0007 wrote the host-neutral
`.planning/config.json` because `knowledge_capture` must be the same block both
hosts read. `pre_execution` is the opposite: it is **host-scoped like the other
15 gates**, so it targets `.planning/config.codex.json`'s `.hooks` object
(D-01/D-19, ADR-0007 point 5's codex-namespacing precedent). Do not copy 0007's
destination.

**jq merge idempotency shape — CORRECTED after review. Use this, not 0007's
shallow form.** See `<merge_safety>` for the full reasoning; this is the
contract:

  idempotency check (LEAF, not the group):
    jq -e '.hooks.pre_execution.plan_review' .planning/config.codex.json >/dev/null

  apply (leaf-level deep merge — never replaces the pre_execution object):
    jq --argjson pe "$PE" '.hooks.pre_execution = ((.hooks.pre_execution // {}) + $pe)'

  rollback (removes only our leaf; drops the parent only if it is then empty):
    jq 'del(.hooks.pre_execution.plan_review)
        | if (.hooks.pre_execution // {}) == {} then del(.hooks.pre_execution) else . end'

`$PE` is the `pre_execution` **object's contents** — i.e.
`{"plan_review": {...}}` — sourced from the installed template
(`$CODEX/skills/setup-codex-agenticapps-workflow/templates/config-hooks.json`,
`.hooks.pre_execution`), never a heredoc'd literal, same single-source rule as
the prose.

**AGENTS.md insert: template extraction, NOT heredoc (D-19).** 0007's Step 2
(its lines 124-147) is the exact mechanism. Its portability comment is
load-bearing, not style:

  "getline-from-file is portable (BSD/macOS awk rejects a multi-line -v assignment)"

A multi-line section CANNOT be passed via `awk -v var="$(cat file)"` on
macOS/BSD awk. The two-pass extract-to-tempfile-then-getline pattern exists
specifically to work around that. Copy the mechanism; change only the heading
regex to the new section's exact heading — read it from `08-04-SUMMARY.md`,
which records the shipped text. A heading-regex mismatch makes the extraction
produce an empty file and the migration a **silent no-op**.

**Checksum command: use `cksum`.** Review flagged the checksum as unspecified
(08-REVIEWS.md, Codex, LOW). Pin `cksum` — POSIX, present on macOS and Linux
alike, unlike `sha256sum` (absent on macOS, where it is `shasum -a 256`) or
`md5`/`md5sum` (which differ by name across the two). Every "second run is a
no-op" assertion compares `cksum < file` before and after.

**Harness primitives** — see plan `08-01`'s `<interfaces>`. `assert_check`'s
two-call idiom (pre-condition "not-applied", post-condition "applied") is the
repo's idempotency convention; `test_migration_0007` lines 620-630 show it.

**test_repo_layout** (`run-tests.sh:776`) is a flat `for f in ... ; do` list of
paths asserted to exist, plus symlink regression guards at the end.

<merge_safety>
## The config merge, corrected — read this before touching Step 1

Cross-AI review raised five HIGH findings against this plan's original merge
(08-REVIEWS.md, Codex). Gemini called this plan "a textbook execution" with no
concerns and OpenCode reviewed it only lightly, so these were adjudicated on
their merits rather than treated as outvoted. Four are adopted; the fifth is
deferred. **None of them overrides D-19 — three of them implement it correctly,
because the original shape contradicted D-19's own stated intent ("preserves
existing keys").**

**1. The idempotency check must test the LEAF.** Original:
`jq -e '.hooks.pre_execution'`. That is true as soon as *any* pre-execution gate
exists — so on an install that already has a sibling pre-execution gate but no
`plan_review`, the migration silently skips and the install never gets the gate,
while reporting success. Test `.hooks.pre_execution.plan_review`.

**2. The merge must be leaf-level, not shallow.** Original:
`.hooks += {pre_execution: $pe}`. `+=` on `.hooks` preserves `.hooks`'s *other
groups* (`pre_phase`, `per_task`, …) — which is what the original fixture proved
— but it **replaces the entire `pre_execution` object**, deleting any sibling
gate inside it. Today `pre_execution` has one member so nothing is lost; the
moment core spec adds a second pre-execution gate, this migration deletes it on
every install that has it. Use
`.hooks.pre_execution = ((.hooks.pre_execution // {}) + $pe)`, which merges at
the leaf and preserves siblings. The `// {}` handles the first-run case where
`pre_execution` does not exist at all.

**3. Rollback must remove only our leaf.** Original: `del(.hooks.pre_execution)`,
destructive to siblings for the same reason. Remove
`.hooks.pre_execution.plan_review`, then drop the now-empty parent only if it is
empty — so a rollback on an install with a sibling gate leaves that gate bound.

**4. The fixture must actually be able to catch 2 and 3.** The original fixture
carried a pre-existing gate under a *different top-level group*
(`post_phase.spec_review`) plus a foreign top-level key — mirroring 0007's
`"host": "claude"` trick. That proves `.hooks += {...}` preserves other groups,
which was never in doubt, and **cannot catch a replaced `pre_execution` object**.
Task 1's fixture must carry a pre-existing `.hooks.pre_execution.other_gate` and
assert it survives BOTH apply and rollback. Keep the original foreign keys too —
they still guard their own regression.

**5. DEFERRED — the update skill's multi-hop chain.** Review found that
`update-codex-agenticapps-workflow` selects pending migrations **once**, from the
initial project version: a project at 0.4.0 selects 0007 but not 0008, applies
0007, lands at 0.5.0, and then fails the final target-version check — so the
supported upgrade chain is untested and broken. **This is a real defect and it is
NOT this plan's work.** It lives in a different skill, and criterion 7 asks only
that `run-tests.sh` pass with an idempotent `test_migration_0008`. Do **not**
repair it here, do **not** add 0.4.0→0.5.0→0.6.0 end-to-end fixtures, and do
**not** widen this migration's `from_version` to paper over it. It is recorded in
08-CONTEXT.md's Deferred Ideas ("Multi-hop migration chain in update skill") and
in ADR-0009's follow-ups. This migration's supported floor stays 0.5.0 → 0.6.0,
which is where every live project already sits after 0007.
</merge_safety>
